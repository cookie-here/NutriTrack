2. Table Definitions
Field Name Data Type Constraints Description
USERS Table
id INTEGER PRIMARY KEY, AUTO_INCREMENT Unique user identifier
email VARCHAR(255) UNIQUE, NOT NULL User email address for login
hashed_password VARCHAR(255) Encrypted password
full_name VARCHAR(255) User's full name
user_type ENUM Type: pregnant, newParent, trying
BABIES Table
id INTEGER PRIMARY KEY, AUTO_INCREMENT Unique baby identifier
user_id INTEGER FOREIGN KEY, NOT NULL Reference to USERS table
name VARCHAR(255) NOT NULL Baby's name
date_of_birth DATE NOT NULL Baby's date of birth
gender ENUM Gender: male, female, other
GROWTH_RECORDS Table
id INTEGER PRIMARY KEY, AUTO_INCREMENT Unique record identifier
baby_id INTEGER FOREIGN KEY Reference to BABIES table
date DATE Measurement date
weight_kg FLOAT Baby's weight in kilograms
height_cm FLOAT Baby's height in centimeters
3. Entity Relationships
USERS (1) ↔ BABIES (N)
One user can have multiple babies. Each baby belongs to one user.
USERS (1) ↔ GROWTH_RECORDS (N)
One user can have multiple growth records for tracking across all babies.
BABIES (1) ↔ GROWTH_RECORDS (N)
One baby can have multiple growth records tracking their development over time.
USERS (1) ↔ REMINDERS (N)
One user can have multiple reminders for vaccines and appointments.
BABIES (1) ↔ REMINDERS (N)
One baby can have multiple reminders for their specific healthcare milestones.
USERS (1) ↔ PARTNERS (N)
One user can have multiple partner relationships (co-parents or family).
PARTNERS (Self-Referencing)
Links two users together as partners using user_id and partner_id fields.
USERS (1) ↔ EMERGENCY_CONTACTS (N)
One user can have multiple emergency contacts.
USERS (1) ↔ NOTES (N)
One user can create multiple notes for personal health tracking.
Legend and Notation
PK - Primary Key (unique identifier for each record)
FK - Foreign Key (reference to another table)
1:N - One-to-Many relationship
Solid Box - Main entity tables
Dashed Box - Reference/static tables
Arrow - Indicates relationship direction
Important Notes:
  • All timestamps include created_at and updated_at fields (managed by application)
  • Foreign keys enforce referential integrity and maintain data consistency
  • VACCINES and FEEDING tables are reference tables containing static data shared across users
  • User authentication provider information supports OAuth and traditional login methods
  • All user data is isolated per user_id ensuring multi-tenancy and data privacy
